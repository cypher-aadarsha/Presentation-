Part 1

Concurrency Control

Ensuring data integrity when multiple transactions run simultaneously.

Why Do We Need Concurrency Control?

To prevent data corruption from interleaving operations (concurrency anomalies).

Without Control (Chaos)

Analogy: Two people editing the same sentence in a shared document at once. The final result is gibberish.

With Control (Order)

Analogy: The document "locks" the sentence for the first editor, making the second one wait. The result is coherent.

Technique 1: Two-Phase Locking (2PL)

A pessimistic approach that locks data before using it to ensure serializability.

Phase 1: Growing Phase

Transaction acquires all the locks it needs. It cannot release any locks in this phase.

Phase 2: Shrinking Phase

Transaction releases its locks. It cannot acquire any new locks in this phase.

Growing Phase --> Lock Point --> Shrinking Phase

Variants of 2PL

Basic 2PL can suffer from cascading rollbacks. These variants prevent it.

Strict 2PL

Holds all exclusive (write) locks until the transaction commits or aborts. Prevents other transactions from reading uncommitted data.

Rigorous 2PL

Holds all locks (read and write) until the transaction commits or aborts. Easiest to implement and is used in most commercial databases.

The Problem with 2PL: Deadlocks

A cycle of transactions waiting for locks held by each other.

[T1] LOCK(Account_A)

[T2] LOCK(Account_B)

[T1] Tries to LOCK(Account_B) -> Waits for T2

[T2] Tries to LOCK(Account_A) -> Waits for T1

Result: A deadly embrace! Neither can proceed.

Solution: Deadlock detection (via wait-for graphs) or prevention (timeouts). One transaction is chosen as a victim and rolled back.

Lock Granularity: A Balancing Act

What size of data item do we lock? This is a trade-off between concurrency and overhead.

Fine Granularity (e.g., Row Lock)

Pro: High Concurrency
Con: High lock management overhead

Medium Granularity (e.g., Page Lock)

A common compromise.

Coarse Granularity (e.g., Table Lock)

Pro: Low overhead
Con: Very low concurrency

Technique 2: Timestamp Ordering

Uses timestamps to determine the serial order of transactions. Older transactions get priority.

  • Each transaction is assigned a unique timestamp when it starts: TS(T).
  • Every data item has a Read-Timestamp and a Write-Timestamp.
  • The protocol checks TS(T) against the data's timestamps to ensure the order is maintained.
  • If a transaction is too "late" to read or write, it is aborted and restarted with a new timestamp.

Technique 3: Multiversion Concurrency Control (MVCC)

"Readers don't block writers, and writers don't block readers."

Instead of overwriting data, the database creates a new version of the data item upon a write. Each transaction sees a consistent "snapshot" of the database as it existed when the transaction began.

[T1 starts] Reads Account Balance = $100 (Version 1)

[T2 starts] Reads Account Balance = $100 (Version 1)

[T2 writes] Account Balance = $120 (Creates Version 2)

[T1 reads again] Still sees Account Balance = $100 (Version 1)

Technique 4: Validation (Optimistic)

Assumes conflicts are rare. "Act first, ask for permission later."

READ PHASE
→
VALIDATION PHASE
→
WRITE PHASE
Transactions execute freely in a private workspace. Before committing, the validation phase checks if the transaction's operations would violate serializability. If the check passes, it writes to the database. If not, it's rolled back.

Technique 5: Snapshot Isolation

A popular MVCC implementation. Guarantees a transaction sees a consistent snapshot of the DB.

  • A transaction operates on a database snapshot taken at the moment it started.
  • Prevents "dirty reads" and "non-repeatable reads".
  • A transaction will only commit if no other committed transaction has modified the same data it has modified. This is the "first-committer-wins" rule.

The Danger of Snapshot Isolation: Write Skew

Scenario: On-Call Doctor Rota

Rule: At least one doctor must be on call.

  1. Two doctors, Alice and Bob, are on call. They both want to take the day off.
  2. [T-Alice starts] Reads rota, sees 2 doctors on call. (Ok to take off)
  3. [T-Bob starts] Reads rota, sees 2 doctors on call. (Ok to take off)
  4. [T-Alice writes] Sets her status to OFF. Commits.
  5. [T-Bob writes] Sets his status to OFF. Commits.

Result: Both transactions succeed because they didn't modify the *same* data. But the database is now in an inconsistent state (0 doctors on call). This is a write skew anomaly.

Part 2

Database Recovery

Restoring the database to a consistent state after a failure.

Core Recovery Concepts

Failures are inevitable. The goal is to ensure the ACID properties (Atomicity, Consistency) are maintained.

The Transaction Log

The most critical component. It's a sequential, append-only file on stable storage that records all database modifications before they happen on disk (Write-Ahead Logging).

<T1, start>
<T1, Account_A, old_val, new_val>
<T1, commit>

Recovery 1: Deferred Update

"Don't write it until you're sure." Updates are only written to the database disk after a transaction commits.

Recovery Process: NO-UNDO / REDO

  • Since uncommitted transactions never wrote to the disk, there's nothing to UNDO.
  • The recovery system scans the log and uses the log records to REDO the operations of any committed transactions.

Recovery 2: Immediate Update

"Write it now, but log it first." Updates are written to the database disk as they happen.

Recovery Process: UNDO / REDO

  • The recovery system must UNDO the changes of any transactions that started but did not commit before the crash.
  • It must also REDO the changes of any transactions that did commit, to ensure their changes are persistent.

Update Techniques: Comparison

FeatureDeferred UpdateImmediate Update
Recovery ActionREDO onlyUNDO and REDO
ComplexitySimpler recoveryMore complex recovery
OverheadLonger commits (writes happen at end)Faster commits (writes happen during)

Optimizing Recovery: Checkpoints

Scanning the entire log on recovery is slow. Checkpoints are used to limit the work.

Periodically, the system performs a checkpoint:

  1. Forces all modified data from memory to disk.
  2. Writes a <checkpoint> record to the log.

On recovery, the system only needs to scan the log forward from the last checkpoint, dramatically speeding up the process.

Advanced Recovery: ARIES

A widely adopted algorithm for recovery based on Immediate Update.

ANALYSIS PHASE
→
REDO PHASE
→
UNDO PHASE
Analysis: Scans log forward from last checkpoint to identify active transactions.
REDO: Repeats all operations (even for aborted transactions) to bring DB to its state at the moment of the crash.
UNDO: Scans log backward to revert the operations of uncommitted transactions.

Recovery 3: Shadow Paging

An elegant, no-log recovery technique. It never overwrites active data pages.

When a write occurs, a copy of the page is made (a "shadow page"). The original is untouched. A "current" page table points to the modified pages.

If the transaction commits, the "current" page table becomes the new master. If it aborts or crashes, the "current" page table is simply discarded.

DB Pointer → Shadow Page Table (Original)

Transaction Pointer → Current Page Table (Modified)

Shadow Paging: Crash Scenario

Online Banking Transfer

  1. State: The main DB pointer points to the Shadow Page Table.
  2. Transaction Starts: A new Current Page Table is created.
  3. Write 1: User A's balance is debited. A copy of that data page is created, and the Current Page Table is updated to point to this new page.
  4. CRASH! The system fails before the transaction commits.
  5. Recovery: On restart, the system discards the Current Page Table. The main DB pointer never moved from the original Shadow Page Table. The database is instantly consistent. No UNDO needed.

Recovery 4: Backup & Recovery

For catastrophic failures (e.g., disk destruction), local logs and shadow pages are useless.

Full Backup

A complete copy of the entire database.

Differential Backup

Copies all changes made since the last full backup.

Transactional Log Backup

Copies the transaction log records.

Scenario: Data Center Fire

Recovering an E-commerce Database

Timeline: A fire destroys the main database server at 3:30 PM.

  1. Step 1: Provision a new server in a different location.
  2. Step 2: Restore the last off-site full backup (Sunday 2:00 AM).
  3. Step 3: Restore the last off-site differential backup (Today 1:00 PM).
  4. Step 4: Apply all subsequent transaction log backups, which are shipped off-site every 15 minutes, up to the 3:15 PM backup.
  5. Result: About 15 minutes of data loss (Recovery Point Objective), but the database is recovered.

Advanced Recovery: Point-in-Time (PITR)

Scenario: Accidental `DROP TABLE`

Timeline: A developer accidentally drops the main `users` table at 10:05:30 AM.

  1. Step 1: Restore the last full/differential backup.
  2. Step 2: Instead of applying all log backups, use a database command like RESTORE ... WITH STOPAT = '10:05:29'.
  3. Result: The database is restored to the exact second before the disastrous command was executed. This is possible because of the continuous stream of transaction log backups.

Summary & Key Takeaways

Concurrency Control

It's a trade-off between performance and correctness. Pessimistic methods like Rigorous 2PL are safe but can cause deadlocks. Optimistic methods like MVCC are fast for reads but can introduce subtle anomalies like write skew.

Database Recovery

The Write-Ahead Log is the foundation for recovering from most failures, optimized by checkpoints. For catastrophic failures, a disciplined backup and log shipping strategy is the only solution, enabling powerful features like PITR.
